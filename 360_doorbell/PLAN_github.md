# 花 300 块买了个门铃，逆向 300 小时还没看到加密视频

> 又名：一个消费级门铃的加密层数比网银还多。它想保护什么？邻居的脚步声吗。

---

## 它只是一台门铃

360 智能可视门铃。618 打折 300 块买的。淘宝包邮。

它的加密体系：

- **libjiagu 360 加固** — 反调试、反 Frida、反注入，会主动杀进程。五个 syscall 劫持才按住
- **TLS 双向证书锁定** — Conscrypt 定制，MITM 直接放弃
- **0x8009 八阶段信令握手** — 比 TCP 三次握手多了五步，每步带 MD5 签名
- **RC4 信令加密** — 对，2015 年就退役的 RC4。key 每 session 随机生成
- **ChaCha20 视频流加密** — 15 个 32B key 云端下发，每 12 小时轮换
- **四路网络并发** — UDP P2P + TCP 护盘 + HTTP API + 电话信令，缺一条就瘸
- **session 随机 wrapper 协商** — 每连接随机生成 2B 前缀区分视频/音频/电话
- **双层 TLV 嵌套编码** — 外层明文 JSON 包内层 RC4 密文
- **自定义 MD5 sign 公式** — 不是标准 MD5，参数排序规则靠逆向猜

## 安全架构赏析（嘲讽向）

**RC4 + 自定义 MD5 排序 = 掩耳盗铃。**

字面意义的「掩耳盗铃」。RC4 的 session key 在 0x8009 握手中通过那个自定义排序的 MD5 协商出来。但没有人需要破解 RC4——hook 住 RC4 加解密前的明文入口就行。libjiagu 再强，也拦不住 JNI_OnLoad 之前下断点。

**双层 TLV（外层 JSON + 内层 RC4）= 半桶水心态。**

为了兼容老版本服务器，外层留了 JSON 明文字段——设备 SN、时间戳全在。这意味着 RC4 的随机数种子空间等于被明文泄露了一半。厂商以为自己很聪明：外层给旧服务器看，内层给新服务器看。实际上给了逆向工程师一个完美的已知明文攻击入口。

**ChaCha20 15 个 key 云端下发 = 把钥匙和锁放在同一个信封里。**

key 通过 RC4 加密的信令通道传输。破解了 RC4 信令层，15 个 ChaCha20 key 就全裸了。这不是纵深防御——这是把两扇门装在同一个门框上，第一扇门的钥匙挂在第二扇门上。

**但是——libjiagu 是真的能打。**

得替 360 工程师说句公道话。libjiagu 反加固不是纸糊的：mmap(0x0) 预占、ptrace 劫持、exit/_exit/abort 全部劫持、kill/raise/tkill/tgkill 五个信号全拦、mprotect 强制 R+X——五层 syscall 劫持少一个都跑不起来。spawn 模式必死，attach 要卡四秒窗口。这个反调试壳是认真做了的。

可能他们的想法很简单：APP 启动→libjiagu 保住前门→后面随便传。毕竟谁闲着没事去劫持五个 syscall 来逆向一台门铃呢？

……好吧，是我们。

**总结：**

```
厂商心态：
  RC4 → 够旧够偏门，不会有人实现 → 安全
  MD5 排序 → 乱序一下没人猜得到 → 安全
  双层TLV → 外面明文兼容、里面密文保护 → 安全
  ChaCha20 → 现代密码学！→ 安全
  四路并发 → 你连哪路干啥都分不清 → 安全

逆向工程师：
  RC4 → Frida hook 5 分钟抓 key
  MD5 → 调几个请求比对参数就出来了
  TLV外层 → 谢谢你的 SN 和时间戳
  ChaCha20 → 谢谢你在 RC4 信令里传 key
  四路并发 → tshark 10 秒自动分类
```
